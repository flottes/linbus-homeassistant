# Motorhome LIN Bus Bridge

Reverse-engineered LIN bus data from a motorhome, read by an ESP32 acting
as LIN master and decoded on a Raspberry Pi. Decoded values are published
to Home Assistant over MQTT.

Hardware: ESP32 D1 Mini + WomoLIN interface (TJA1021T transceiver).
Bus: LIN @ 19200 baud, enhanced checksum (LIN 2.x). The devices use
CI-Bus, which runs on LIN.

The ESP32 is a "dumb" timing-accurate frontend. It executes single LIN
transactions on command over USB serial. All decoding and scheduling
happens in Python on the Pi.

## My Bus Devices

- Dometic RC 10.4T 70 fridge
- Votronic VBCS 60/40/430 Triple

## Bus overview

Responding IDs found by scanning 0x00..0x3F:

| ID   | Device                        | Length | Notes                     |
|------|-------------------------------|--------|---------------------------|
| 0x0C | Dometic RC 10.4T 70 fridge    | 8      | always awake              |
| 0x16 | Votronic VBCS battery current | 4      | sleeps without charging   |
| 0x19 | Votronic (unknown)            | 4      | always 00 00 00 00 so far |
| 0x20 | Votronic solar MPP regulator  | 5      | sleeps without charging   |

The whole Votronic LIN node sleeps when no charge source is active
(no mains, no B2B, no sun). IDs 0x16/0x19/0x20 stop answering together
and return when a charge source wakes the node. The Dometic node stays
awake continuously.

## Reverse-engineered frames

### Votronic solar MPP regulator
```
ID  : 0x20
data: 6F 05 00 00 40
       |  |  |  |  |
      PL PM PH PX  |          = Power, 32-bit little-endian, value / 47500 = W
                  ST          = Charger status
                               0x40 = offline / sleep
                               0x44 = bulk
                               0x4D = bulk+absorb
                               0x5D = absorb
                              b2: 1 = online / bulk, 0 = offline/sleep
                              b3: 1 = absorb
                              b6: 1 (always)
```
Power verified as ~0 W with panels dark (display showed 0 W).
This node is the solar path only; the mains/B2B charge current does not
appear here, only the status byte reflects charging.

### Votronic battery current
```
ID  : 0x16
data: 91 01 18 00
       |  |  |  |
       CL CH  |  |            = Current, 16-bit little-endian, value / 10 = A
             ST              = Status byte
                              b3: 1 (always when running)
                              b4: 1 = actively charging (0x18 active / 0x08 idle)
```
Current verified against the device display: raw 0x0191 = 401 -> 40.1 A.
Value tracks the display in real time. No sign/negative current observed;
value drops to 0 rather than going negative.

### Dometic RC 10.4T 70 fridge (compressor, CI-Bus)
```
ID  : 0x0C
data: 62 03 02 05 00 00 80 90
       |  |  |  |  |  |  |  |
      B0 B1 B2 B3 B4 B5 B6 B7
      |  |  |                 B1 = cooling level, 1..5 (raw value)
      |  |  B2                B2 b5 (0x20): 1 = door open, 0 = closed
      B0                      B0 b6 (0x40): 1 = unit on
      |                       B0 b3-b4     : operating mode (2-bit field)
      |                                      00 = performance
      |                                      10 = silent
      |                                      11 = boost
                           B7 b7 (0x80): 1 = compressor / running
```
Constant bytes in all observed frames: B3=0x05, B6=0x80 (b7 always set).
B4/B5 always 0x00.

Cooling level and mode are retained when the unit is switched off
(only the two "on" bits B0 b6 and B7 b7 clear).

Verified by toggling each function and watching which bit/byte moved:
- door open/close  -> B2 b5 only
- level 1..5       -> B1 = 1..5
- mode change      -> B0 b3-b4
- power on/off     -> B0 b6 and B7 b7

Open / unconfirmed:
- B2 b1 (0x02) toggled occasionally; likely compressor-cycle / cooling
  demand, not yet confirmed.
- Actual interior temperature not found in this frame. B4/B5 stay 0.
  May not be broadcast on 0x0C at all.

## ESP32 firmware (lin_bridge_esp32.cpp)

Command-driven LIN frontend. Line-based text protocol over USB serial
@115200. The ESP32 only does LIN timing; it does not decode.

LIN break is generated by briefly switching the UART to half baud rate
and sending 0x00, which produces a dominant phase long enough for a valid
break without touching UART registers.

Commands:
```
SCAN            scan all IDs 0x00..0x3F once
READ <id>       poll one ID once            (hex, e.g. READ 0C)
POLL <id> <ms>  poll one ID repeatedly       (debug; STOP ends it)
STOP            stop repeated poll
BAUD <n>        set LIN baud rate (9600 / 19200)
HELP            command list
```

Response lines (machine- and human-readable):
```
RESP <id> <pid> <len> <b0..bn> <crc> <classic|enh|bad>
EMPTY <id>          no answer
OK <text>           info / confirmation
ERR <text>          error
SCANEND <count>     scan finished
```

Pins (ESP32 UART1, freely mappable): RX = GPIO16, TX = GPIO17.
The first two response bytes are the ESP32's own echo (sync 0x55 + PID),
which the transceiver loops back; they are skipped. The last byte is the
checksum.

## Python client (lin_bridge.py)

Runs on the Pi. The Pi is the scheduler: it decides which ID to poll when
and sends single READ commands to the ESP32. Decoding lives in `decode()`.

Modes:
```
scan            list all responding IDs once
read <id>       query one ID once
monitor         multi-ID scheduler, output to console
mqtt            operating mode: poll all IDs, publish to Home Assistant
inspect <id>    live dissector for one ID (see below)
```

Examples:
```
python3 lin_bridge.py --port /dev/lin-bridge scan
python3 lin_bridge.py --port /dev/lin-bridge inspect 0C
python3 lin_bridge.py --port /dev/lin-bridge mqtt
```

Dependencies: `pip install pyserial paho-mqtt`

### inspect mode

Live reverse-engineering tool. Polls one ID continuously and renders a
fixed dashboard that updates in place (no scrolling log). Shows:

- Raw bytes, with bytes that just changed highlighted.
- Bit matrix: every byte as a row, bits 7..0 as columns. Green = 1,
  grey = 0. Bits that have changed at any point since start turn yellow,
  with a per-byte change mask. This is how digital states (e.g. door
  open/closed) are isolated: toggle the function and watch which bit
  turns yellow.
- Number interpretations: each byte as signed int8; every adjacent byte
  pair as 16-bit LE and BE, unsigned and signed, with /10 and /100
  scalings; 32-bit LE/BE over B0..B3 with /47500.
- Byte activity: min..max over the whole run, showing which bytes are
  live (span > 0) and which are constant.

The yellow "ever changed" marks are sticky, so a bit that reacts even
once stays flagged. Restart inspect to reset the history.

### monitor / mqtt modes

`monitor` runs the multi-ID scheduler and prints decoded values. Each ID
has its own interval (see `POLL_PLAN`); the scheduler is non-blocking so a
slow ID does not hold up a fast one.

`mqtt` runs the same scheduler but publishes to Home Assistant with MQTT
auto-discovery. Configuration (broker, credentials, topic prefix) is at
the top of the script.

Availability smoothing: a source is only marked "unavailable" in Home
Assistant after several consecutive failed polls (`AVAILABILITY_MAX_FAILS`),
not after a single missed frame. This prevents flapping and handles the
Votronic sleep behaviour. A last-will message marks everything offline if
the bridge process dies.

Published entities (auto-discovered under one device):

| Entity                  | Type          | Source |
|-------------------------|---------------|--------|
| Votronic current (A)    | sensor        | 0x16   |
| Votronic charging       | binary_sensor | 0x16   |
| Votronic solar power (W)| sensor        | 0x20   |
| Votronic charger status | sensor        | 0x20   |
| Fridge cooling level    | sensor        | 0x0C   |
| Fridge mode             | sensor        | 0x0C   |
| Fridge power on         | binary_sensor | 0x0C   |
| Fridge compressor       | binary_sensor | 0x0C   |
| Fridge door             | binary_sensor | 0x0C   |

## Deployment (systemd)

`lin-bridge.service` runs the `mqtt` mode on boot.

```
sudo cp lin-bridge.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now lin-bridge
journalctl -u lin-bridge -f
```

A udev rule pins the ESP32 to a stable device path regardless of ACM
enumeration order:
```
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="55d4", \
  ATTRS{serial}=="574D043110", SYMLINK+="lin-bridge"
```
Adjust the serial to your board (`dmesg` after plugging in shows it).

The service holds the serial port. Stop it before running `inspect` or
`scan` by hand: `sudo systemctl stop lin-bridge`.
